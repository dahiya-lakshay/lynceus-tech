# Lynceus — Real-Time, Explainable Financial Fraud Detection

Named for the Argonaut famed for eyesight so sharp he could see through walls and into the earth
itself. That's the design goal: a fraud engine that sees through deception and can point at exactly
what it saw.

```
raw transaction
   ├─▶ Feature store lookup (per-account behavioral baseline)
   ├─▶ Isolation Forest (anomaly score)
   ├─▶ Model leaderboard winner: Random Forest vs Hist Gradient Boosting (fraud probability)
   ├─▶ Rules engine (independent deterministic checks)
   └─▶ SHAP (why)
```

Two entry points, one scoring function: a REST API for on-demand checks, and a Kafka consumer for
a continuous event stream. Both call the exact same `score_transaction()`.

## What's in here

- **A feature store** (`backend/app/ml/account_profile.py`) — per-account behavioral baselines
  (spending mean/std, known devices, known locations) computed once from history, looked up at
  inference time, with a cold-start fallback to global stats for accounts never seen before. This
  is the same online/offline split every real fraud system has to solve.
- **A model leaderboard** — Random Forest and Histogram Gradient Boosting are cross-validated
  (5-fold, stratified) and the better one is promoted to production automatically. Both entries are
  reported on the dashboard, not just the winner.
- **Threshold optimization** — the decision boundary is the F1-maximizing point on the held-out
  precision-recall curve, not an assumed 0.5.
- **A rules engine** (`backend/app/ml/rules.py`) that runs independently of the ML model: excessive
  login attempts, amount far exceeding account balance, an unrecognized device *and* location
  together, statistical outliers against the account's own history. Real fraud systems are almost
  never ML-only, and these flags are reported as corroborating evidence, not blended into the score.
- **No label leakage** — the anomaly score is deliberately excluded from the classifier's own
  features (it's what the label was thresholded from; including it lets the model trivially
  memorize the label instead of learning the pattern). Held-out metrics are the real thing:
  precision 0.85 / recall 0.88 / F1 0.86 / ROC-AUC 0.99 on data the model never trained on.
- **An event-driven streaming path** (`backend/streaming/`) — a Kafka producer simulates a live
  transaction feed, a consumer scores each one through the same pipeline the API uses and publishes
  results to a "scored" topic. The frontend's Live Feed section reflects this when it's running, and
  gracefully falls back to on-demand scoring when it isn't.
- **CI/CD** (`.github/workflows/`) — backend: lint, train, test, Docker build, optional Render
  deploy-hook trigger. Frontend: install, build check (Vercel handles actual deploys via its GitHub
  integration).
- **Tests** (`backend/tests/`) — pipeline unit tests and FastAPI smoke tests via `TestClient`.

**Honest limitation, stated rather than hidden:** this dataset has no verified fraud label —
"fraud" here means "flagged as anomalous by Isolation Forest at a 5% contamination rate." That's a
reasonable, common approach when no labeled data exists, but the system should be read as a
risk-prioritization tool for analyst review, not a certified fraud determination. The frontend's
footer says this plainly rather than overclaiming.

## Running locally

### Backend

```bash
cd backend
python -m venv venv && source venv/bin/activate   # venv\Scripts\activate on Windows
pip install -r requirements.txt

python -m app.ml.train          # writes to backend/models/ and backend/outputs/
uvicorn app.main:app --reload --port 8000
```

Interactive API docs: `http://localhost:8000/docs`.

### Frontend

```bash
cd frontend
npm install
cp .env.example .env    # VITE_API_URL=http://localhost:8000 for local dev
npm run dev
```

Visit `http://localhost:5173`. Press **⌘K** / **Ctrl+K** for the command palette.

### Full stack with Docker

```bash
docker compose up backend frontend           # request/response app, Live Feed scores on demand
docker compose --profile streaming up        # adds Kafka + producer + consumer for the real event stream
```

## Retraining on your own data

Replace `backend/data/transactions.csv` with a file using the same columns, then rerun
`python -m app.ml.train`. The feature store, model leaderboard, threshold, and SHAP explanations all
regenerate from the new data automatically.

## API reference

**`POST /api/predict`** — score a single transaction (`AccountID` optional — omit it to simulate an
account never seen during training and see the feature store's cold-start path).

**`GET /api/stats`** — held-out metrics, cross-validation, model leaderboard, precision-recall curve,
global feature importance, most anomalous transactions in the training set.

**`GET /api/stream/recent`** / **`GET /api/stream/next`** — powers the Live Feed. Reads from the
Kafka consumer's output when the streaming profile is running; otherwise scores a random sample
transaction on demand.

**`GET /api/health`** — liveness check.

## Deploying

### Backend → Render

1. Push to GitHub.
2. Render → **New → Blueprint**, point at this repo (picks up `backend/render.yaml`), or create a
   **Web Service** manually with root directory `backend` and the Docker runtime.
3. Set `ALLOWED_ORIGINS` to your Vercel frontend URL.
4. The Docker build trains the model at build time (see `backend/Dockerfile`), so the deployed API
   starts with artifacts already in place.
5. Optional: add a Render **Deploy Hook** URL as the `RENDER_DEPLOY_HOOK_URL` GitHub Actions secret
   so pushes to `main` trigger a redeploy automatically (see `.github/workflows/backend-ci.yml`).

### Frontend → Vercel

1. Import the repo, set the project root to `frontend`.
2. Add environment variable `VITE_API_URL` = your Render backend URL.
3. Deploy — `vercel.json` handles the Vite preset and SPA rewrites.
4. Back on Render, add this Vercel URL to `ALLOWED_ORIGINS` and redeploy the backend.

### Streaming (optional, for the full event-driven demo)

The Kafka profile in `docker-compose.yml` is meant for local/demo use or a VM — running Kafka on
Render's free tier isn't practical. If you want the streaming path live in production, the cleanest
option is a managed Kafka service (Confluent Cloud, Upstash Kafka, AWS MSK) — point
`KAFKA_BOOTSTRAP_SERVERS` at it for the backend, producer, and consumer, and deploy the consumer as
its own long-running Render worker service (not the Docker build path used for the API).

## Tech stack

kafka-python, pandas, joblib, uvicorn, pytest.
**Frontend:** React, Vite, Tailwind CSS, Recharts, Axios.
**Infra:** Docker, docker-compose (Kafka in KRaft mode), GitHub Actions.
